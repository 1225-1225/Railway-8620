## 📊 一、计算机基础与数据结构

### Q1：时间复杂度和空间复杂度，常见时间复杂度等级
**答：**  
- 时间复杂度：度量算法执行时间随输入规模n增长的趋势。  
  - O(1) 常数：哈希表查找  
  - O(log n) 对数：二分查找  
  - O(n) 线性：遍历数组  
  - O(n log n) 线性对数：归并排序、快速排序平均  
  - O(n²) 平方：冒泡排序、嵌套循环  
- 空间复杂度：度量额外内存占用随n增长的趋势。递归调用栈深度常为O(n)，原地排序为O(1)。

### Q2：数组和链表的区别及适用场景
**答：**  
| 比较维度  | 数组                           | 链表                                    |
| --------- | ------------------------------ | --------------------------------------- |
| 内存分配  | 连续内存，大小固定或动态扩容   | 非连续，节点在堆上动态分配              |
| 随机访问  | O(1) 索引直接访问              | O(n) 需要遍历                           |
| 插入/删除 | O(n) 需要移动元素              | O(1) 修改指针即可（已知位置时）         |
| 缓存友好  | 好，连续内存预读               | 差，节点分散                            |
| 适用场景  | 频繁读、随机访问、大小相对固定 | 频繁插入删除、不确定大小、实现队列/栈等 |

### Q3：哈希表工作原理及冲突解决
**答：**  
- 原理：通过哈希函数将键映射到数组下标，理想O(1)查找。  
- 冲突处理：  
  1. 链地址法：每个槽位存链表/红黑树，冲突元素追加进去。Java HashMap采用此方式。  
  2. 开放寻址法：线性探测、二次探测、双哈希等，查找下一个空槽位。

### Q4：栈和队列的区别及应用
**答：**  
- 栈：后进先出（LIFO），只允许在栈顶操作。应用：函数调用栈、括号匹配、撤销操作。  
- 队列：先进先出（FIFO），在队尾插入、队首删除。应用：BFS、消息队列、任务调度。  
Python中`list`可做栈，`collections.deque`高效做队列。

## ⚙️ 二、后端开发与API设计

### Q5：RESTful API设计原则及项目实践
**答：**  
- 资源导向：用URL定位资源，如`/users/{user_id}`  
- HTTP方法语义化：GET(查)、POST(增)、PUT(全量改)、PATCH(部分改)、DELETE(删)，GET/PUT/DELETE需幂等  
- 无状态：服务器不保存会话，每个请求携带全部鉴权信息（JWT）  
- 统一接口，可缓存  
- 项目实践（FastAPI + JWT鉴权）：  
  `POST /auth/register` 注册用户  
  `POST /auth/login` 登录返回JWT（OAuth2 Password表单）  
  `POST /chat/stream` SSE流式聊天接口  
  `GET /chat/sessions` 获取会话历史列表（按日期分组）  
  `GET /chat/sessions/{thread_id}` 获取单次会话详情  
  鉴权通过请求头 `Authorization: Bearer <token>` 传递，后端用依赖注入 `get_current_user()` 统一验证。  
- 注意：本项目车次查询和路线规划**不是通过REST API暴露的**，而是封装在LangGraph Agent的工具函数中，由Agent根据用户意图自动调用——这也是一种架构选择：将业务逻辑留给Agent编排，API层只负责通信。

### Q6：JWT原理、优势及项目鉴权实现
**答：**  
- 结构：Header（算法类型）.Payload（用户id、过期时间等，不存敏感信息）.Signature（对前两部分的签名防篡改）  
- 优势：无状态，服务器不需session存储，便于扩展；跨域友好；移动端适用。  
- 实现：  
  1. 登录成功，后端用密钥签发JWT返回客户端。  
  2. 客户端存于localStorage/Cookie，后续请求header中携带。  
  3. 后端中间件验证签名和有效期，解析出用户信息注入上下文。  
- 防泄露：HTTPS传输，短期有效，可配合Refresh Token旋转，考虑HttpOnly Cookie防XSS。

### Q7：FastAPI 与 Spring Boot 对比
**答：**  
- FastAPI：基于Python异步（ASGI），自动生成OpenAPI文档，轻量，性能极高，适合AI服务、高IO并发、微服务原型。  
- Spring Boot：Java企业级框架，生态完善（Spring全家桶），依赖注入、AOP管理，内置Tomcat，适合大型复杂业务系统。  
我的项目选FastAPI是因为其对异步流式响应的天然支持，以及与LangChain生态的紧密整合。

## 🤖 三、AI应用核心

### Q8：RAG（检索增强生成）原理与项目实现
**答：**  
- 原理：从外部知识库检索相关文档片段，注入Prompt作为上下文，让LLM基于事实回答，减少幻觉。  
- 项目经历了两个阶段，但**核心架构一致**——都是 `@tool` 装饰器注册工具函数，由 LangGraph Agent 中的 LLM **自主决策是否调用**，不需要人工编排。变化的只是 `retriever_tool` 函数**内部的检索实现**：  

**阶段一：自建 Chroma 向量库（早期）**  
  1. 文档处理：使用 `RecursiveCharacterTextSplitter` 将 150+ 篇铁路文档分块（chunk_size=500, overlap=77），MD5 去重。  
  2. 向量化：阿里百炼 text-embedding-v4（DashScopeEmbeddings），存入 Chroma SQLite 持久化。  
  3. 检索：`retriever_tool` 内部调用 `Chroma.similarity_search()`，**进程内直接查询本地 SQLite**，无需额外服务。  
  4. 痛点：分块策略需反复调优、索引维护麻烦、增删文档要手动重建。  

**阶段二：迁移到 RAGFlow（当前）**  
  1. 文档写入：`ragflow_migrate.py` 批量上传到 RAGFlow，平台内部完成解析、分块和向量化。  
  2. 检索：`retriever_tool` 内部改为调用 `RAGFlowClient.search()`→ HTTP POST 请求 RAGFlow 的 `/api/v1/retrieval` 接口。  
  3. 好处：减少了自建维护成本，同时保留了切换弹性。  

- 整体调用链路：用户输入 → FastAPI 调用 LangGraph Agent.stream() → Agent **自主决定**是否调 `retriever_tool` → 检索结果回传给 LLM → LLM 整合知识生成回答 → FastAPI 流式 SSE 返回前端。  
- 如果面试官追问 Chroma 实践，可以从分块策略（为什么选 500/77）、embedding 选型（text-embedding-v4 对比 BGE）、相似度阈值调优等角度展开。

### Q9：LangChain与LangGraph在Agent开发中的作用
**答：**  
- LangChain：提供Chains（串联调用）、Memory（对话记忆）、Tools（外部工具）、Agents（决策调用工具）等组件，简化LLM应用开发。  
- LangGraph：基于图的状态机框架，将Agent流程建模为节点和条件边，支持循环、分支、多角色协作。相比线性Chain，更适合复杂Agent。  
- 我的项目：用LangGraph构建多工具Agent（共4个工具——`retriever_tool`知识库检索、`query_train_info`车次查询、`query_trains_by_route`路线车次查询、`generate_route_map`地图生成），Agent通过`create_agent`函数组装，使用`SqliteSaver`持久化对话检查点。能自主判断用户意图，按需调用工具，并将工具结果整合为最终回答。后端支持流式（SSE）和非流式两种调用模式。

### Q10：Folium交互地图在项目中的应用
**答：**  
- 项目需要为用户展示车次途经站点的可视化路线图。  
- 实现方案（简化版）：用Folium（Leaflet.js的Python封装）生成交互式HTML地图，读取`station_coords.json`中的站点经纬度坐标，将相邻经停站用直线连接（`folium.PolyLine`），始发站用绿色图标、终点站用红色方格旗、中间站用蓝色圆点，支持鼠标悬停查看站名和缩放。  
- 文件保存到`data/maps/`目录，通过FastAPI静态文件挂载 `/maps` 路径访问。  
- 后续可优化方向：引入NetworkX构建全国铁路拓扑图，用A*算法做真实轨道最短路径搜索，替代目前的直线连接方式。我自己也掌握A*的原理（`f(n)=g(n)+h(n)`），可以根据业务需求把这条路走通。

### Q12：LoRA微调技术
**答：**  
- 原理：冻结预训练权重，在Transformer注意力层旁路注入低秩矩阵AB（秩r远小于原维度），仅训练A和B，参数增量极少。  
- 优势：参数高效（<1%），显存需求低，可灵活切换模块。  
- 应用场景：将通用大模型（如Qwen、Llama）适配垂直领域（铁路术语、调度规则），或定制风格。  
- 我掌握其原理，将来如需提升铁路知识理解深度，可用来微调领域模型。

## 🧩 四、项目深度剖析与工程化

### Q13：项目中遇到的最大挑战及解决
**答：**  
**挑战一：从自建 Chroma 向量库迁移到 RAGFlow 平台**  
- 背景：最初使用 **Chroma + langchain_chroma** 自建向量检索，用 `RecursiveCharacterTextSplitter` 分块（chunk_size=500, overlap=77）配合 MD5 去重，embedding 用的是阿里百炼 text-embedding-v4（DashScopeEmbeddings）。`retriever_tool` 作为一个 `@tool` 注册在 LangGraph Agent 中，内部直接调用 `Chroma.similarity_search()` 进程内查询本地 SQLite，**无需额外服务**。但需要自行处理文档解析、分块策略调优、Chroma SQLite 索引持久化维护等问题，每次增删文档都要手动重建向量库。  
- 方案：  
  1. 调研接入 **RAGFlow**——它提供开箱即用的文档解析、分块和向量管理能力，且支持 REST API 对接。  
  2. 编写 `ragflow_migrate.py` 将 150+ 篇铁路文档批量上传到 RAGFlow。  
  3. 封装 `RAGFlowClient`（`agent/ragflow_client.py`），通过统一接口对接检索，支持 top_k 和相似度阈值配置。  
  4. 通过 `reset_service_singletons()` 支持单例重置，保证单元测试时 RAGFlow 依赖可被 Mock 隔离。  
- 效果：减少了自建维护工作量，检索效果由 RAGFlow 的内部优化保障，同时保留了切换到其他方案（如 Elasticsearch + 自定义 embedding pipeline）的灵活性。  
- 遗留痕迹：旧代码仓库中仍保留 `mytools/retriever_test.py`（Chroma 检索测试脚本）和 `data/vector_database/` 目录，可以从这些细节说明项目确实经历了 Chroma 阶段。

**挑战二：Agent多工具调用的状态管理**  
- 现象：依赖工具间数据传递，状态流转复杂，调试困难。  
- 方案：  
  1. 在LangGraph中清晰定义State对象和节点输入输出。  
  2. 使用LangGraph可视化工具检查状态图。  
  3. 关键节点/工具调用前后打印详细日志。  
  4. pytest写单元测试单独验证工具和Agent子流程。

## 📝 面试小抄·速记卡片（线上面试前看一遍）

### 🔥 数据库（MySQL）
```
▸ 事务ACID: 原子性A、一致性C、隔离性I、持久性D
▸ 隔离级别（由低到高）: 
  ① 未提交读(脏读) → ② 提交读(不可重复读) → ③ 可重复读(MySQL默认, 幻读) → ④ 串行化
▸ MVCC: 多版本并发控制，快照读不加锁，当前读需要加锁
▸ Next-Key Lock: 可重复读级别下防止幻读 = Record Lock + Gap Lock
▸ 索引B+Tree: 多叉平衡树，叶节点存数据/指针，有序，适合范围查询
▸ 聚簇索引: InnoDB主索引，叶子存整行数据
▸ 辅助索引: 叶子存主键值，查询需回表
▸ 最左前缀: 联合索引遵循最左匹配原则
▸ 避免索引失效: 不在索引列上做运算/函数，不隐式类型转换
▸ Explain 重点看: type(ALL全表→range→ref→const), rows(扫描行数), key(使用索引)
▸ 缓存穿透: 查不存在数据→布隆过滤器/空值缓存
▸ 缓存雪崩: 大量key同时过期→随机过期时间/多级缓存
▸ 缓存击穿: 热点key过期→互斥锁/永不过期
```

### ⚡ Redis
```
▸ 5种数据类型: STRING, LIST, SET, HASH, ZSET(有序)
▸ 过期删除: 定期删除+惰性删除
▸ 淘汰策略: allkeys-lru(常用), volatile-lru, noeviction
▸ 持久化: RDB(快照) 和 AOF(日志)
▸ 分布式锁: SETNX key value + EXPIRE (或 RedLock)
▸ 缓存一致性: 先更新DB再删缓存 (Cache Aside Pattern)
```

### 🌐 HTTP / 网络
```
▸ HTTP无状态: 服务器不保存会话信息
▸ GET vs POST:  GET安全幂等可缓存/参数URL有长度限制; POST非安全非幂等/参数body
▸ 状态码: 200OK / 301永久重定向 / 302临时 / 304未修改 / 400坏请求 / 401未认证 / 403禁止 / 404未找到 / 500服务器错误 / 502坏网关 / 503服务不可用
▸ HTTPS: HTTP + SSL/TLS，混合加密(非对称交换密钥+对称加密传输)
▸ 长连接: HTTP/1.1默认Keep-Alive，复用TCP
▸ TCP三次握手: SYN → SYN+ACK → ACK
▸ 四次挥手: FIN → ACK → FIN → ACK
```
